# File system as a service

When an application uses a file system, it's outsourcing several distinct concerns to it. The most fundamental is **naming**: without a file system, you would have to address your data by raw block numbers on a storage device. The file system gives you a hierarchical namespace with human-readable names (like `/home/user/notes.txt`), and it's responsible for translating those names into physical locations. Closely related to this is the **abstraction over hardware**: an application reading a file doesn't know or care whether the underlying storage is a spinning disk, an SSD, an NVMe drive, a USB stick, a network share, or even RAM. The file system presents a uniform "named stream of bytes" interface on top of all of them.

The second big service is **persistence with structure**. Data written to a file outlives the process that wrote it and survives reboots â€” but unlike just dumping bytes onto a disk, the file system tracks where each file's bytes live, how big the file is, how to grow or shrink it, and how to find it again by name later. This includes managing free space (allocation, deallocation, fragmentation) and storing **metadata** â€” timestamps, ownership, size, type â€” alongside the data itself.

Then there's **random access**: applications can read or write at any offset within a file (`lseek` to byte 12345, write 100 bytes, jump back, read), and files can grow and shrink dynamically. This is something raw block devices technically support, but the file system handles all the bookkeeping.

**Concurrency and sharing** are services too. Multiple processes can open the same file simultaneously, and the file system (together with the OS) provides mechanisms to coordinate them: advisory locks, atomic appends, atomic renames, and so on. **Access control** belongs in this bucket as well â€” Unix-style permissions (owner/group/other Ă— read/write/execute), ACLs, and extended attributes let the file system enforce who can do what.

A modern file system also provides **crash consistency** guarantees of various strengths. If the power dies mid-write, you don't want your file system metadata to be corrupted such that the whole tree becomes unreadable. Techniques like journaling (ext4) or copy-on-write (btrfs, zfs) give different trade-offs between performance and the strength of guarantee. Note that crash consistency for the *file system's own structures* is much stronger than crash consistency for *your application's data* â€” getting your data durable requires `fsync()` and friends, which I'll come back to.

Finally, file systems often layer in **performance services** (caching, read-ahead, write-back buffering â€” although on Linux most of this lives in the kernel page cache rather than in the file system itself) and **advanced features** that vary widely by FS: snapshots, compression, encryption, deduplication, sparse files, hard links, symbolic links, subvolumes, quotas, etc.

# APIs

You're right that there's no single universal standard, but on Linux the situation is actually quite uniform: essentially every file system exposes the **POSIX file API** to applications, mediated by the kernel's VFS layer (more on that in the next section). So from an application's point of view, ext4, xfs, btrfs, tmpfs, NFS, and a FUSE-based filesystem all look the same. The differences are in performance characteristics, feature availability, and edge-case semantics â€” not in the API surface.

The POSIX API can be grouped roughly as follows. **Path-based operations** take a string path and act on the named file: `open`, `creat`, `unlink` (delete), `rename`, `stat` (get metadata), `access`, `chmod`, `chown`, `truncate`, `mkdir`, `rmdir`, `symlink`, `link` (hard link), `readlink`. **File-descriptor-based operations** take an integer fd returned by `open` and act on the already-opened file: `read`, `write`, `lseek` (seek to offset), `pread`/`pwrite` (read/write at a specific offset without changing the seek pointer, useful for concurrency), `close`, `fstat`, `ftruncate`, `fsync` (force data + metadata to disk), `fdatasync` (force just data). **Directory operations** are `opendir`, `readdir`, `closedir`, `rewinddir`. **Memory-mapped I/O** lets you map a file into your address space and access it as memory: `mmap`, `munmap`, `msync`. **Locking** is via `flock` or `fcntl` with `F_SETLK`. There's also a more modern family of `*at` syscalls (`openat`, `unlinkat`, `renameat`, etc.) that take a directory fd plus a relative path, which avoids race conditions and lets you work relative to a directory without `chdir`.

Beyond POSIX, Linux has its own extensions. **`inotify`** and **`fanotify`** let you watch files and directories for changes. **`io_uring`** is the modern high-performance asynchronous I/O interface, replacing the older and somewhat broken `aio` family. **`statx`** is a richer version of `stat` exposing things like creation time and btrfs/xfs-specific attributes. Extended attributes have their own API: `getxattr`, `setxattr`, `listxattr`, `removexattr`. And then each file system can expose specific functionality through `ioctl` calls â€” for example, btrfs snapshot creation is an ioctl on the parent directory's fd. So while the *common* surface is POSIX, file-system-specific features do leak through ioctls and a handful of Linux-specific syscalls.

One important conceptual point: most file system APIs are **not standardized for durability semantics**. The standard says `write()` returns once data has been handed to the kernel â€” it does not say it's on disk. To get a durability guarantee, you must call `fsync()` (or `fdatasync()`), and even then the exact guarantees depend on the file system, the kernel version, and the storage device's own behavior. This is a famously sharp edge â€” see "fsyncgate" for a fun read.

# Relation between file systems and the Linux kernel

On Linux, file systems are not standalone services â€” they are kernel components, and they sit in a layered architecture that's worth walking through top to bottom.

At the top is the **system call interface**. When your application calls `open("/etc/passwd", O_RDONLY)`, glibc wraps that in a syscall that traps into the kernel. The syscall handler for `open` lives in the kernel's VFS code, not in any specific file system.

The **VFS (Virtual File System)** is the kernel layer that makes all file systems look the same to userspace. It defines a set of abstract objects: the `superblock` (represents a mounted file system instance), the `inode` (represents a file â€” the metadata, not the name), the `dentry` (represents a name in the directory tree, and is heavily cached), and the `file` (represents an open file, i.e. an instance of an inode being used by some process). Each of these has a corresponding **operations table** â€” `super_operations`, `inode_operations`, `dentry_operations`, `file_operations` â€” which is essentially a vtable of function pointers. When you call `read()` on an fd, the VFS looks up the `file_operations.read` pointer for that fd and dispatches to it. This is how the same syscall ends up calling completely different code depending on whether the underlying file is on ext4, xfs, NFS, or `/proc`.

Below the VFS sit the **individual file system implementations**: ext4, xfs, btrfs, f2fs, tmpfs, procfs, sysfs, NFS, ceph, FUSE, and many more. Each of them implements those operations tables in its own way. They register themselves with the kernel as available file system types, and when someone runs `mount -t ext4 /dev/sda1 /mnt`, the VFS asks the ext4 module to populate a superblock for that device and grafts the resulting tree into the unified namespace at `/mnt`.

The unified namespace is itself a key concept. Linux has a single tree rooted at `/`, and different file systems are **mounted** at different points in that tree. There is no drive letter â€” `/`, `/home`, `/boot`, `/proc`, `/tmp`, and `/mnt/usb` might all be different file systems, possibly of different types, and the application doesn't have to know or care.

Sitting alongside the file systems is the **page cache**. When you `read()` from a file, the kernel checks if those pages are already in memory; if so, it copies from cache without touching the disk. When you `write()`, the data goes into the page cache and is written back to the underlying device asynchronously by kernel threads (unless you opened with `O_SYNC` or call `fsync`). The page cache is unified across all file systems and is also what backs `mmap`. This is why "free memory" on a Linux box is usually misleading â€” most of it is cache that can be reclaimed at any time.

Below the file system implementations is the **block layer**, which handles I/O queueing, scheduling (deadline, mq-deadline, bfq, none), and request merging for actual block devices. Below that are the **device drivers** (NVMe, AHCI/SATA, virtio-blk, MD/RAID, dm/LVM/dm-crypt) and finally the hardware. A single read from an encrypted, RAID-mirrored ext4 partition on an NVMe SSD might pass through the page cache, ext4, dm-crypt, MD, the block layer, the NVMe driver, and the device â€” all transparent to the application.

Two things make Linux's file system architecture especially powerful. First is the **"everything is a file"** philosophy: pipes, sockets, devices, processes (`/proc`), kernel knobs (`/sys`, `/sys/fs/cgroup`), and even some kernel events (`signalfd`, `timerfd`, `eventfd`) are exposed as file descriptors and accessed through `read`/`write`/`poll`. This means the same code that processes a file can often process a network socket or a device. Second is **FUSE** (Filesystem in Userspace), which lets a regular userspace program implement a file system. The FUSE kernel module registers as a normal VFS file system, but it forwards every operation to a userspace daemon over a special character device. This is how things like `sshfs`, `s3fs`, encfs, and many others work â€” they get to look like regular file systems without anyone needing kernel privileges.
